

## Giới thiệu về Lập trình Hướng Đối tượng (Object-Oriented Programming - OOP)

### Tại sao cần học OOP?

Khi các dự án code trở nên phức tạp hơn (như dự án Coffee Machine ở Day 15), code của chúng ta bắt đầu trở nên rối rắm và khó quản lý. Lý do:

- Một function thay đổi nhiều biến khác nhau
- Các mối quan hệ giữa functions và variables ngày càng nhiều
- Logic code trở nên "spaghetti-like" - rối như mớ bòng bong
- Rất khó theo dõi và nhớ những gì đang xảy ra trong code


### Lập trình Thủ tục (Procedural Programming)

**Đặc điểm:**

- Thiết lập các thủ tục (procedures) hoặc hàm (functions) để thực hiện công việc cụ thể
- Một thủ tục dẫn đến thủ tục khác
- Máy tính chủ yếu làm việc từ trên xuống dưới, nhảy vào function khi cần
- Là một trong những paradigm lập trình sớm nhất (ví dụ: Fortran, COBOL)

**Vấn đề:**
Khi độ phức tạp và số lượng mối quan hệ tăng lên, code trở nên cực kỳ khó quản lý và gây nhầm lẫn.

### Lập trình Hướng Đối tượng (OOP) - Giải pháp

OOP giúp duy trì các mối quan hệ đơn giản trong code trong khi vẫn có thể viết các dự án phức tạp hơn.

**Ví dụ minh họa: Xe tự lái (Self-driving Car)**

Thay vì xử lý toàn bộ dự án phức tạp như một khối, ta chia nhỏ thành các module:

- **Camera Module**: Theo dõi và nhận diện các vật thể trên đường
- **Lane Detection**: Phát hiện làn đường, biết khi nào cần rẽ hay đậu xe
- **Navigation**: Xác định lộ trình đến đích
- **Fuel Management**: Quản lý nhiên liệu, tự động sạc pin khi cần

**Lợi ích của việc modularization:**

- Nhiều team có thể làm việc đồng thời trên các module khác nhau
- Tăng năng suất đáng kể
- Các module có thể **tái sử dụng** (reusable) cho dự án khác (ví dụ: drone delivery)
- Không cần viết lại code cho chức năng tương tự


### Ví dụ thực tế: Nhà hàng

**Mô hình một người (Procedural Programming):**

- Một người phải làm tất cả: lễ tân, phục vụ, đầu bếp, người dọn dẹp
- Giới hạn quy mô - chỉ phục vụ được vài khách
- Rất khó quản lý khi mở rộng

**Mô hình có team (OOP):**

- Mỗi người có vai trò riêng: waiter, chef, cleaner
- Mỗi người được đào tạo chuyên môn cho vai trò của mình
- Bạn chỉ cần làm manager - chỉ đạo công việc
- Không cần quan tâm chi tiết cách họ thực hiện (waiter biết cách phục vụ, chef biết cách nấu)
- Có thể mở rộng quy mô nhà hàng dễ dàng

**Áp dụng vào lập trình:**

- Đơn giản hóa các mối quan hệ trong code
- Làm cho code có thể mở rộng (scalable) cho dự án lớn và phức tạp hơn
- Mỗi "đối tượng" (object) biết cách thực hiện công việc của mình
- Chúng ta chỉ cần "quản lý" và gọi các đối tượng khi cần


### Tóm lại

OOP giúp:

- **Chia nhỏ** (modularize) code thành các phần dễ quản lý
- **Tái sử dụng** (reuse) code hiệu quả
- **Mở rộng** (scale) dự án dễ dàng hơn
- **Đơn giản hóa** mối quan hệ giữa các thành phần
